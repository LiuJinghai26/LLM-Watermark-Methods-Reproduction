# 对比实验审查

本文档审查当前 KGW、SWEET、EWD 对比实验是否足以体现算法差别，并给出建议增加的数据类别和样本数量。

## 当前实验结论

当前 `results/opt_self_ppl` 的 20 条本地测试可以证明完整链路可运行：OPT-1.3B 生成、三种水印检测、PPL 计算、指标汇总和作图均能完成。

但这组结果不适合作为最终对比结论的主要依据，原因是：

- 样本数只有 20 条，统计波动较大。
- 数据只有 C4 RealNewsLike 新闻类文本，文本自然度高、候选 token 熵通常较高。
- 当前结果中 KGW、SWEET、EWD 的 `auroc` 和 `tpr_at_5pct_fpr` 都为 1.0，检测性能已经饱和，难以体现算法间的真实差异。
- EWD 在本项目中是检测器而不是独立生成算法，因此应表述为“KGW 生成 + EWD 熵加权检测”，不能和 KGW、SWEET 完全并列为三个生成算法。

当前 20 条测试仍可放在报告中作为本地可复现实验或烟测结果，但正式对比建议使用更多样本和更多数据类型。

## 三种算法差异应如何体现

KGW 的核心差异是对所有可计分 token 使用绿名单 logit 偏置，并用绿名单比例计算 z-score。它适合展示基础水印强度和文本质量损失。

SWEET 的核心差异是只在高熵 token 上嵌入和检测水印。它的优势应主要体现在低熵或混合熵场景中：减少对确定性 token 的干预，可能牺牲部分可计分 token 数量，但改善质量或鲁棒性。

EWD 的核心差异是检测时按熵加权。它应主要通过低熵、混合熵文本中的检测分数变化体现优势；在高熵新闻文本上，EWD 和 KGW 的差别可能较小，或者只表现为 z-score 数值变化。

因此，仅用高熵新闻数据很容易得到“都能检测出来”的结论，却很难说明 SWEET 和 EWD 的设计价值。

## 建议增加的数据类别

建议至少保留一类自然语言数据，再增加一类低熵或结构化数据。

| 数据类别 | 用途 | 建议样本数 |
| --- | --- | --- |
| C4 RealNewsLike 新闻文本 | 高熵自然语言主实验，和论文设置接近 | 本地 200 条；云端 500 条更稳 |
| HumanEval 或 MBPP 代码/类代码文本 | 低熵、强结构场景，用于体现 SWEET/EWD 的熵相关差异 | HumanEval 164 条，或 MBPP 200-500 条 |
| 模板化/结构化文本 | 低熵补充实验，避免只依赖代码数据 | 100-200 条 |

如果必须严格保持同一生成模型，代码类数据也应继续使用 OPT-1.3B 生成。这样生成代码质量未必好，但仍能作为低熵/结构化文本的水印检测压力测试。若改用代码模型，则需要在报告中单独说明该实验不再满足“同一模型”约束。

## 建议实验规模

最低本地验证：

- C4 RealNewsLike 50 条。
- 代码或结构化数据 50 条。
- 用 OPT-1.3B 自身算 PPL，作为本地近似质量指标。

报告推荐规模：

- C4 RealNewsLike 200 条。
- HumanEval 164 条，或 MBPP 200 条。
- 若显存允许，在云端使用 OPT-6.7B 作为 PPL/oracle 模型。

更稳健的规模：

- C4 RealNewsLike 500 条。
- HumanEval 164 条。
- MBPP 500 条或模板化结构数据 200 条。

## 指标和图表建议

每个数据类别应单独汇总，不建议只给总平均值。建议保留以下指标：

- 检测指标：`auroc`、`tpr_at_5pct_fpr`、平均 z-score、绿名单比例。
- 嵌入强度：平均可计分 token 数、SWEET 跳过比例、EWD 权重和。
- 质量指标：水印文本 PPL、相对 Plain 的 PPL 增量。
- 效率指标：生成耗时、检测耗时，作为补充。

建议图表：

- z-score 分布箱线图，区分 human/plain 与 watermarked。
- ROC 曲线；若曲线饱和，可补充 TPR@1%FPR。
- PPL 与 z-score 的质量-可检测性散点图。
- 不同数据类别下的可计分 token 数或熵均值对比图。

## 最终判断

当前实验可以证明代码链路和模型本地运行没有问题，但不足以充分体现 KGW、SWEET、EWD 的算法差异。正式报告建议至少增加一类低熵或结构化数据，并将样本数从 20 条提升到 200 条级别。若只保留当前 C4 20 条结果，结论应限定为“本地小样本验证”，不宜作为算法优劣判断。
